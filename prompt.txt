# CyberKavach — System Logic, Pipeline & Enhanced Prompt

## PART 1 — UNDERLYING LOGIC & SYSTEM THINKING

### Core Architectural Logic

The system operates on three foundational principles:

**1. Everything is a workflow.** Every action in the club — creating an event, generating a certificate, posting on social media — must pass through a defined chain of accountability. The platform encodes these chains into code rather than relying on WhatsApp threads and verbal confirmations.

**2. Permissions are data, not code.** Rather than hardcoding if-else permission checks, roles and permissions live in the database. This means adding a new role, restricting a feature, or granting temporary elevated access requires a database update, not a deployment.

**3. Every action leaves a trail.** The audit log is not optional. Every insert, update, delete, approval, rejection, and login is recorded with the user ID, timestamp, IP address, and the before/after state. This transforms the platform from a tool into an institutional memory.

---

### Technology Stack Logic (PHP/MySQL)

Since you're using PHP + MySQL + phpMyAdmin:

- **Backend:** PHP 8.2+ with PDO for database access, no raw `mysqli_*` calls
- **Frontend:** Vanilla HTML5, CSS3 (custom design system, no Bootstrap), JavaScript ES6+ with select libraries (Alpine.js for reactivity, Chart.js for analytics, QRCode.js)
- **Database:** MySQL 8.0+ via phpMyAdmin, with foreign keys, indexes, and stored procedures for complex operations
- **Sessions:** PHP sessions + JWT tokens stored in httpOnly cookies
- **File Storage:** Local disk with Cloudinary API integration for media
- **PDF:** FPDF or TCPDF library for certificate generation
- **Email:** PHPMailer with SMTP (Gmail / SendGrid)
- **QR:** endroid/qr-code or chillerlan/php-qrcode
- **Real-time:** Polling-based with AJAX (optional: PHP Ratchet for WebSockets)
- **Security:** PHP's built-in `password_hash()`, prepared statements, CSRF tokens, CSP headers

### MODULE 1 — MULTI-LEVEL APPROVAL ENGINE

This is the most critical module. It must be bulletproof.

**Approval Flow Architecture:**

```
Request Created by User
    ↓
Level 1: Department Coordinator (Tech/Content/Social) — 24hr SLA
    ↓ [if approved]
Level 2: Student Coordinator — 48hr SLA
    ↓ [if approved]
Level 3: Faculty Coordinator — 72hr SLA
    ↓ [if approved]
Request Fulfilled (action triggered automatically)
```

**PHP Implementation Requirements:**

- `ApprovalService::createRequest()` — creates approval request + first step
- `ApprovalService::processStep()` — records decision, advances or closes chain
- `ApprovalService::escalate()` — called by cron when SLA exceeded
- `ApprovalService::getTimeline()` — returns full step history with remarks
- Each approval type has a config: `APPROVAL_TYPES` constant defining levels, SLA, and auto-action on completion
- Cron job runs every 15 minutes to check SLA violations and send reminder emails
- Escalation sends notification to next-level approver with context
- All approval actions are audit-logged

**Request Types:**
- `event_create` — event goes live after final approval
- `venue_booking` — lab/room request
- `budget_request` — financial allocation
- `social_post` — post published on approval
- `content_publish` — blog post goes live
- `certificate_authorize` — bulk cert generation enabled
- `external_collaboration` — inter-club or external partner

### MODULE 2 — CERTIFICATE SYSTEM

**Certificate Generation Pipeline:**

```
1. Faculty/Student Coord uploads PNG/PDF template
2. Template editor: drag-drop field zones (name, event, date, ID)
3. Field mapping stored in certificate_field_maps table
4. Upload CSV of recipients OR link to attendance data
5. Validation: check all fields present, name normalization
6. Queue batch job (PHP queue table or Redis-equivalent)
7. FPDF/TCPDF renders each cert: places fields at (x%, y%) coordinates
8. Each cert gets unique ID: CK-YYYY-EVENTCODE-NNNNNN
9. SHA-256 hash of (cert_id + user_id + event_id + secret_key) stored as verification_hash
10. QR code generated pointing to /verify/{cert_id}
11. QR embedded in certificate image
12. Batch zipped + uploaded to Cloudinary + Google Drive
13. Individual emails sent via PHPMailer queue
14. Generation log with success/failure per recipient
```

**Verification Page (/verify/{id}):**
- Lookup by cert ID or verification hash
- Display: holder name, event, date, issuing authority
- Show tamper status (hash match = valid)
- Display downloadable verification report PDF
- No login required (public endpoint)

```
Coordinator creates attendance session (generates unique QR token)
    ↓

    ↓

    ↓

    ↓
Server validates: token valid + not expired + user registered + not duplicate
    ↓
Record inserted, live dashboard updates via polling
    ↓
Late flag set if after event start time + grace period
```

### SECURITY ARCHITECTURE (NON-NEGOTIABLE)

Every single one of these must be implemented:

**Authentication Security:**
- Passwords: `password_hash($password, PASSWORD_BCRYPT, ['cost' => 12])`
- Sessions: regenerate ID on login (`session_regenerate_id(true)`)
- Cookies: `HttpOnly`, `Secure`, `SameSite=Strict`, `Path=/`
- JWT: signed with HS256, expiry 15 minutes, refresh token 30 days in httpOnly cookie
- OTP: 6-digit, SHA-256 hashed before storage, 10-minute expiry, max 3 attempts

**CSRF Protection:**
- Token generated per session: `bin2hex(random_bytes(32))`
- Embedded as hidden field in every HTML form
- Sent as `X-CSRF-Token` header in every AJAX request
- Validated in middleware before any POST/PUT/DELETE handler
- Token rotated after each use

**Input Security:**
- All user input passes through `Sanitizer::clean()` before use
- HTML content (rich text) passed through HTMLPurifier with strict allowlist
- All DB queries use PDO prepared statements — zero string interpolation in SQL
- File uploads: MIME-type detection via `finfo_file()`, extension whitelist, rename with UUID, store outside web root

**Rate Limiting:**
- Login: 5 attempts per IP per 15 minutes → 30-minute lockout
- OTP: 3 requests per email per hour
- Registration: 3 accounts per IP per day
- API endpoints: 60 requests per minute per authenticated user
- Certificate generation: 1 batch per event per 5 minutes

**HTTP Security Headers (.htaccess):**
```apache
Header always set Content-Security-Policy "default-src 'self'; script-src 'self' cdn.jsdelivr.net; img-src 'self' data: res.cloudinary.com; font-src 'self' fonts.gstatic.com"
Header always set X-Frame-Options "DENY"
Header always set X-Content-Type-Options "nosniff"
Header always set Referrer-Policy "strict-origin-when-cross-origin"
Header always set Permissions-Policy "geolocation=(self), camera=(self)"
Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
```

**Audit Logging — Every action logged:**
```php
Audit::log(
    action: 'certificate.generate',
    module: 'certificates',
    record_type: 'certificate_batch',
    record_id: $batchId,
    old_value: null,
    new_value: ['event_id' => $eventId, 'count' => $count]
);
```

### SCALABILITY CONSIDERATIONS

Design decisions that ensure the system scales:

- **ULID primary keys** for all user-facing entities (sortable, globally unique, no enumeration)
- **Soft deletes** on users, events, teams (never hard-delete)
- **Metadata JSON columns** on approvals and events for extension without schema changes
- **Queue table** for async jobs (cert generation, bulk email) — replaceable with Redis/Beanstalk
- **Percentage-based certificate field positioning** — works at any resolution
- **Module isolation** — each module's code is self-contained, can be extracted as a microservice
- **Multi-club ready** — add `club_id` to all entity tables and filter everywhere (architecture supports it from day one)
